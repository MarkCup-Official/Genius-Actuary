# Genius Actuary 开发任务清单

## 1. 文档目标

本文档基于 [ProjectProposal.md](M:\Documents\Practice\Genius-Actuary\Agent\ProjectProposal.md) 的策划案整理为可执行的 AI Agent 开发任务清单，适用于基于 MCP 思维进行模块化开发、并行协作与阶段验收。

目标：

- 将 MVP 拆解为明确、清晰、可交付的任务
- 为每个任务定义输入、输出、依赖与完成标准
- 明确开发顺序，避免实现时反复返工
- 为最终联调、验收、测试提供统一标准

---

## 2. 开发原则

### 2.1 总体原则

- 以 MVP 闭环优先，不提前建设非目标能力
- 所有模块优先输出结构化协议，不依赖隐式上下文
- 推理、联网、计算、图表、持久化、展示必须解耦
- 先打通单会话主链路，再做优化与增强
- 每一阶段必须有可演示结果和可验证标准
- 后端主要业务逻辑与前端展示层必须分离
- 第一阶段测试统一基于网页端执行
- 第一版 MVP 只交付网页端，但后端接口需为小程序和 App 预留复用能力

### 2.2 MCP 角色定义

本项目中的 MCP 模块按职责划分为：

- `LLM Analysis MCP`：结构化分析、追问生成、结论沉淀、最终报告生成
- `Search MCP / Adapter`：执行 Brave Search 联网任务，返回标准化证据摘要
- `Calculation MCP / Adapter`：执行精确计算，返回公式、参数、结果、误差说明
- `Chart MCP / Adapter`：执行图表生成，返回图表 spec 或可渲染产物

说明：

- 搜索、计算、图表可以是外部 MCP，也可以先由本地适配层封装成统一接口
- Orchestrator 负责决定何时调用哪个 MCP，不允许前端直接编排核心分析链路

## 3. 总体开发顺序

建议按以下顺序开发：

1. Python 后端项目骨架与统一协议
2. 数据模型与持久化
3. Orchestrator 状态机主链路
4. 分析模式与循环分析输入输出协议
5. 动态追问引擎
6. 主要结论文档与上下文治理
7. Search MCP 适配
8. Calculation MCP 适配
9. Chart MCP 适配
10. 最终报告生成模块
11. Web 前端 MVP 交互闭环
12. 观测、缓存、降级与整体联调
13. 网页端验收测试与发布前检查

原因：

- 没有统一协议和状态机，后续 MCP 接入会反复改接口
- 没有结论文档与上下文治理，循环分析无法稳定
- 没有搜索/计算/图表适配层，最终报告能力不完整
- 没有前后端明确分层，后续扩展小程序和 App 会重复建设业务逻辑
- 没有网页端闭环和测试，无法验证 MVP 是否真正可用

## 4. 里程碑

### M1. 核心骨架可运行

完成 Python 后端、Web 前端基础项目结构、核心数据模型、状态机和协议定义。

### M2. 循环分析闭环跑通

支持模式选择、问题输入、追问生成、回答回填、主要结论沉淀。

### M3. MCP 能力接入完成

支持联网搜索、精确计算、图表生成，并纳入主流程。

### M4. 最终报告与前端展示完成

支持用纯净上下文生成最终 Markdown 报告，并在网页端展示图表。

### M5. 稳定性与验收通过

通过网页端主流程联调、异常降级、测试与验收检查。

## 5. 任务清单

以下任务按推荐开发顺序排列。状态默认建议为 `TODO`。

### T01. 建立项目工程骨架

- 目标：初始化 Python 后端、Web 前端、服务层、适配层、共享协议目录结构
- 输出：
  - Web 前端应用骨架
  - Python 后端 API / 服务骨架
  - `orchestrator/`、`adapters/`、`domain/`、`persistence/`、`prompts/`、`shared/` 目录
  - 环境变量模板与启动说明
- 依赖：无
- 完成标准：
  - 本地可启动 Web 前端与 Python 后端基础服务
  - 目录结构可承载后续模块拆分
  - 基础 lint / format / build 命令可运行
  - 前端与后端可独立启动和部署

### T02. 定义领域模型与结构化协议

- 目标：把策划案中的核心对象和 MCP 输入输出协议固化为类型与文档
- 输出：
  - `AnalysisSession`
  - `ClarificationQuestion`
  - `UserAnswer`
  - `EvidenceItem`
  - `CalculationTask`
  - `ChartArtifact`
  - `AnalysisReport`
  - `MajorConclusionItem`
  - `SearchTask`
  - 循环分析输入协议
  - 循环分析输出协议
  - 最终报告输入协议
- 依赖：T01
- 完成标准：
  - 所有关键对象有明确字段定义
  - 所有 MCP 调用有统一 request / response 类型
  - 结构化协议能覆盖策划案第 5、6、7 章内容
  - 协议不绑定网页端页面结构，支持未来小程序和 App 复用

### T03. 设计数据库与持久化层

- 目标：落地 MVP 所需数据表与仓储接口
- 输出：
  - 数据库 schema
  - repository / DAO 接口
  - 会话、追问、回答、证据、计算结果、图表、报告、结论的增删改查能力
- 依赖：T02
- 完成标准：
  - 可创建并查询完整分析会话
  - 支持按 `session_id` 聚合读取上下文包
  - 支持报告版本与主要结论追加

### T04. 实现 Orchestrator 状态机

- 目标：实现阶段 1、阶段 2、阶段 3 的主流程编排
- 输出：
  - 状态定义：`INIT`、`CLARIFYING`、`ANALYZING`、`READY_FOR_REPORT`、`REPORTING`、`COMPLETED`、`FAILED`
  - 阶段切换规则
  - MCP 调用决策入口
- 依赖：T02, T03
- 完成标准：
  - 能根据会话状态推进下一步动作
  - 能判断继续追问、联网、计算、出报告的时机
  - 状态切换可记录日志并可追踪
  - 编排逻辑仅存在于后端，不放在前端

### T05. 实现分析模式选择器

- 目标：支持单项成本/风险分析与多选项决策参考两种模式
- 输出：
  - 模式枚举与模板
  - 不同模式的输出结构约束
  - 模式初始化逻辑
- 依赖：T02, T04
- 完成标准：
  - 用户创建会话时可明确选择模式
  - 后续 prompt、报告结构、图表建议随模式变化

### T06. 构建 Prompt 模板与上下文分层器

- 目标：按策划案的上下文治理原则构建稳定 prompt 体系
- 输出：
  - 系统层 prompt
  - 单项分析模板
  - 多选决策模板
  - 上下文六层拼装器
  - 截断与注入策略
- 依赖：T02, T04, T05
- 完成标准：
  - 支持循环分析 prompt 组装
  - 支持最终报告纯净上下文组装
  - 可以区分事实、估算、推断、未知项

### T07. 实现初始问题理解接口

- 目标：用户输入问题后，系统能生成第一轮轻量理解结果
- 输出：
  - 问题理解服务
  - 初始结构化问题定义
  - 第一批追问任务生成入口
- 依赖：T04, T05, T06
- 完成标准：
  - 输入一句自然语言问题，可生成结构化 `problem_statement`
  - 结果可进入后续追问阶段

### T08. 实现动态追问引擎

- 目标：让 AI 只提高价值问题，并返回可渲染表单结构
- 输出：
  - 追问生成逻辑
  - 问题优先级排序
  - 跳过/拒答/不确定的处理逻辑
- 依赖：T06, T07
- 完成标准：
  - 每轮追问结果符合协议
  - 每个问题带 `purpose`
  - 能控制轮次和问题数量
  - 用户拒答后能转入保守分析分支

### T09. 实现循环分析上下文包构建器

- 目标：将用户事实、搜索摘要、计算结果、主要结论组装成统一上下文包
- 输出：
  - `context package builder`
  - 未解决不确定项提取逻辑
  - 上下文快照存储
- 依赖：T03, T06, T08
- 完成标准：
  - 上下文包字段与策划案 6.2 一致
  - 不直接拼接全量聊天记录
  - 可复用于多轮循环分析

### T10. 实现主要结论文档机制

- 目标：把每轮新结论增量沉淀为可复用核心资产
- 输出：
  - 主要结论去重与合并逻辑
  - `事实 / 估算 / 推断` 分类存储
  - 结论文档版本化
- 依赖：T03, T09
- 完成标准：
  - 每轮只追加新的高价值结论
  - 重复结论不重复堆积
  - 最终报告可仅依赖该文档与必要计算/图表数据

### T11. 实现循环分析引擎

- 目标：在阶段 2 中生成三类结构化输出：追问、搜索任务、主要结论
- 输出：
  - 循环分析 LLM 调用服务
  - 输出校验器
  - 非法结构修复或重试机制
- 依赖：T06, T09, T10
- 完成标准：
  - 输出包含 A/B/C 三部分
  - 输出字段齐全且结构合法
  - 可为搜索、追问、结论沉淀直接提供输入

### T12. 实现 Search MCP 适配层

- 目标：接入 Brave Search，并统一成标准证据输出
- 输出：
  - 搜索任务执行器
  - 查询重写与结果去重
  - 搜索摘要标准化
  - 来源可信度基础评估
- 依赖：T02, T11
- 完成标准：
  - 输入结构化 `SearchTask` 可得到 `EvidenceItem` 列表
  - 每条摘要包含标题、URL、来源、抓取时间、摘要、事实、可信度
  - 搜索失败时返回可降级结果，而不是中断主流程

### T13. 实现 Calculation MCP 适配层

- 目标：把需要精确求解的问题交给计算模块执行
- 输出：
  - 计算任务标准化
  - 参数校验与单位校验
  - 结果结构化返回
- 依赖：T02, T11
- 完成标准：
  - 支持公式求值、盈亏平衡点、交点、区间等基础计算
  - 结果包含公式、参数、结果、误差或适用说明
  - 可被主要结论和图表直接消费

### T14. 实现 Chart MCP 适配层

- 目标：根据分析或计算结果生成可视化产物
- 输出：
  - 图表任务模型
  - 图表配置生成器
  - 图表 spec / 图片 / 渲染数据输出
- 依赖：T02, T13
- 完成标准：
  - 至少支持柱状图、折线图、雷达图
  - 能区分真实值、估算值、推断值
  - 数据不足时明确返回不可绘制原因

### T15. 实现搜索结果与计算结果注入链路

- 目标：将搜索证据和计算结果回注到上下文和结论文档
- 输出：
  - evidence 注入逻辑
  - calculation summary 注入逻辑
  - basis 引用关系
- 依赖：T10, T12, T13
- 完成标准：
  - 后续轮次可引用既有证据和计算结论
  - 主要结论能标注依据来源

### T16. 实现最终报告生成模块

- 目标：基于纯净上下文生成最终 Markdown 报告
- 输出：
  - 最终报告 prompt
  - 报告生成服务
  - 单项分析报告模板
  - 多选项报告模板
- 依赖：T05, T06, T10, T15
- 完成标准：
  - 不带中间追问文本
  - 可输出结构化 Markdown 报告
  - 报告内容与选定模式一致

### T17. 实现图表与报告汇总模块

- 目标：将最终报告、关键计算结果和图表产物组合成前端可展示结果
- 输出：
  - 报告视图模型
  - 图表引用与排序规则
  - 不确定性与免责声明模块
- 依赖：T14, T16
- 完成标准：
  - 报告与图表可共同展示
  - 图表和结论有清晰对应关系

### T18. 实现前端 MVP 交互层

- 目标：完成网页端用户从提问到查看报告的最小闭环
- 输出：
  - 网页端模式选择页
  - 网页端问题输入页
  - 网页端追问表单页
  - 网页端分析进行中状态页
  - 网页端报告展示页
- 依赖：T05, T08, T17
- 完成标准：
  - 用户可完成一次完整分析流程
  - 支持选项回答、自定义输入、跳过
  - 能查看结构化报告与图表
  - 页面只负责展示与交互，不承载核心业务判断
  - 交互全部通过后端 API 驱动

### T19. 实现日志、追踪与观测

- 目标：建立 MVP 可调试、可复盘的观测基础
- 输出：
  - prompt 版本日志
  - token 用量日志
  - 上下文包快照
  - 搜索/计算/图表调用日志
  - 报告生成链路日志
- 依赖：T04, T11, T12, T13, T14, T16
- 完成标准：
  - 每轮关键动作都可追踪
  - 出错时能快速定位问题发生位置

### T20. 实现缓存与成本优化

- 目标：按策划案中的上下文治理和缓存策略降低调用成本
- 输出：
  - 稳定 prompt 片段缓存
  - 主要结论文档优先缓存
  - 搜索结果缓存
  - 计算结果缓存
  - 图表配置缓存
- 依赖：T06, T10, T12, T13, T14, T19
- 完成标准：
  - 相同输入不会重复触发高成本流程
  - 局部变更可局部重算
  - 缓存命中策略有可观测指标

### T21. 实现异常处理与降级策略

- 目标：保证搜索失败、图表失败、计算失败时系统仍可输出有边界结果
- 输出：
  - 搜索失败降级
  - 计算失败降级
  - 图表失败降级
  - LLM 输出不合法重试或保守回退
- 依赖：T12, T13, T14, T16, T19
- 完成标准：
  - 任一非关键模块失败不应直接导致整次分析崩溃
  - 最终报告会明确标注缺失与影响范围

### T22. 实现安全边界与风险提示

- 目标：避免把系统包装成绝对权威，符合策划案中的产品风险要求
- 输出：
  - 辅助决策免责声明
  - 高影响未知项提示
  - 专业领域边界提示
  - 数据来源与时效提示
- 依赖：T16, T17, T18
- 完成标准：
  - 报告中必须包含假设前提与不确定性说明
  - 涉及政策、法律、签证等场景时有边界提示

### T23. 主流程联调与示例场景验证

- 目标：以网页端为测试载体，对策划案中的典型问题完成端到端联调
- 输出：
  - 海外交换场景联调记录
  - 买车 vs 公共交通场景联调记录
  - 研究生 vs 先工作场景联调记录
- 依赖：T18, T19, T20, T21, T22
- 完成标准：
  - 三个典型场景均能完整跑通
  - 输出结果结构稳定
  - 可复现问题与改进项可记录
  - 联调全部基于网页端完成

### T24. 测试补齐与发布前验收

- 目标：形成可交付的网页端 MVP 验收版本
- 输出：
  - 单元测试
  - 集成测试
  - 端到端测试
  - 验收记录
- 依赖：T23
- 完成标准：
  - 核心用例测试通过
  - 验收项通过率达到发布要求
  - 网页端测试结果满足首版上线门槛

  - 主要结论文档
  - 循环分析输出

### 阶段三：MCP 接入阶段

- 完成任务：T12-T15
- 交付物：
  - 搜索适配
  - 计算适配
  - 图表适配
  - 证据/计算结果注入链路

### 阶段四：用户可见结果阶段

- 完成任务：T16-T18
- 交付物：
  - 最终报告生成
  - 图表与报告展示

### 阶段五：稳定性与发布阶段

  - 观测日志
  - 缓存优化

  - 联调报告
  - 测试与验收报告

## 9. 验收标准

### 9.1 功能验收标准

满足以下条件视为 MVP 功能验收通过：

1. 用户可以选择两种分析模式之一发起会话
2. 用户可以输入自然语言问题并得到第一轮结构化追问
3. 系统每轮都能输出：
   - 需要用户补充的信息
   - 需要联网查找的任务
   - 新的主要结论
4. 用户回答能够被结构化保存，并进入下一轮分析
5. 搜索任务能返回带来源与抓取时间的摘要结果
6. 计算任务能返回结构化且可验证的精确结果
7. 图表任务在数据充分时能生成柱状图、折线图或雷达图
8. 系统能沉淀主要结论文档，且最终报告不直接依赖完整历史聊天
9. 最终报告能按模式输出结构化 Markdown
10. 系统在某个 MCP 模块失败时仍能以降级方式完成分析
11. 网页端可以完整承载首版用户流程

### 9.2 质量验收标准

满足以下条件视为质量验收通过：

1. 不重复追问用户已经回答过的问题，除非信息冲突
2. 报告必须区分事实、估算、推断和未知项
3. 报告必须展示关键假设、不确定性和适用边界
4. 搜索结论必须可追溯到来源 URL
5. 关键计算必须可追溯到公式、参数和结果
6. 图表必须标明单位、维度或数据来源
7. 最终报告应明显比中间轮次更简洁、更稳定
8. 上下文包长度受到控制，不直接膨胀为全量聊天记录

### 9.3 工程验收标准

满足以下条件视为工程验收通过：

1. 核心模块边界清晰，前端不直接耦合外部 MCP
2. 所有结构化协议有类型约束与基础校验
3. 每轮分析链路有日志和快照
4. 搜索、计算、图表调用失败可被捕获并返回明确信息
5. 相同输入具备基础缓存复用能力
6. 关键主流程具备自动化测试覆盖
7. 后端主要业务逻辑使用 Python 实现
8. 前后端解耦，前端替换为小程序或 App 时不需要重写核心编排逻辑

## 10. 测试标准

### 10.1 单元测试标准

以下模块必须具备单元测试：

- 结构化协议校验器
- Orchestrator 状态切换
- 追问优先级排序
- 上下文包构建器
- 主要结论合并与去重
- 搜索结果标准化
- 计算参数校验
- 图表输入校验
- 报告模板拼装
- 降级策略判断

通过标准：

- 核心分支覆盖
- 异常输入有明确断言
- 关键类型转换和协议校验通过

### 10.2 集成测试标准

以下链路必须具备集成测试：

- 创建会话 -> 选择模式 -> 输入问题
- 生成追问 -> 提交回答 -> 构建上下文包
- 循环分析 -> 生成搜索任务 -> 注入搜索结果
- 循环分析 -> 生成计算任务 -> 注入计算结果
- 结论沉淀 -> 生成最终报告
- 图表生成 -> 报告展示聚合

通过标准：

- 每条链路的结构化输入输出合法
- 各模块字段映射一致
- 持久化读写正确
- 前后端接口契约与网页端调用一致

### 10.3 端到端测试标准

至少覆盖以下 3 个真实场景：

1. 是否参加海外交换
2. 是否买车还是继续公共交通
3. 是否申请研究生还是先工作 2 年

每个场景至少验证：

- 可以发起分析
- 至少产生一轮高价值追问
- 至少产生一个联网任务
- 如适用，至少产生一个计算任务
- 最终输出报告
- 如数据充分，至少输出一个图表

通过标准：

- 无阻塞性错误
- 最终结果结构完整
- 降级提示可读且不误导
- 所有 E2E 测试均以网页端为执行入口

### 10.4 回归测试标准

以下改动后必须跑回归：

- Prompt 模板调整
- 上下文包字段调整
- 数据 schema 变更
- 搜索、计算、图表适配器变更
- 报告模板变更

回归通过标准：

- 典型场景输出不出现字段缺失
- 最终报告结构不退化
- 主要结论文档不会异常膨胀

### 10.5 人工验收测试标准

产品和工程共同验收以下体验项：

- 追问是否真正高价值，而不是机械追问
- 搜索结果是否确实补充了外部事实，而不是噪音
- 计算结果是否增强了结论可信度
- 图表是否帮助理解，而不是制造误导
- 报告是否有明确建议、前提条件和不确定性说明
- 在用户跳过回答时，系统是否能给出保守但有用的分析
- 网页端交互是否足够完成首版演示与真实测试

## 12. 建议的首批落地优先级

如果只做第一轮开发，建议先完成以下最小闭环：

1. T01-T11，先打通“问题输入 -> 追问 -> 结论沉淀”的核心循环
2. T12-T13，补上搜索与计算两个最关键的外部能力
3. T16-T18，完成最终报告与网页端展示
4. T19-T21，补上必要的日志、缓存、降级
5. T23-T24，做典型场景联调与验收

图表能力 T14 可以与 T16 并行，但不建议早于搜索和计算优先实现。

首版上线说明：

- 首个 MVP 上线范围仅为网页端
- 测试、联调、验收全部以网页端为准
- 小程序端和手机 App 端进入后续版本规划

- 自测通过
- 关键日志可观测
- 相关依赖任务未被破坏
