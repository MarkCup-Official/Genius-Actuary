# Genius Actuary 项目策划案

## 1. 项目概述

### 1.1 项目名称
Genius Actuary

### 1.2 项目定位
这是一个面向个人用户的 AI 决策分析 Agent。

核心目标不是直接给出“拍脑袋式建议”，而是把一个决策、一个计划、一个行动推进到底后可能产生的：

- 显性成本
- 隐性成本
- 短期风险
- 长期风险
- 机会成本
- 不确定性来源
- 可替代方案
- 决策建议

系统化梳理出来，并通过结构化分析辅助用户做出更理性的判断。

### 1.3 典型场景

- 我是否应该参加大学大三的海外交换？
- 我是否应该辞职创业？
- 我是否应该买车而不是继续打车或坐地铁？
- 这个副业项目推进 6 个月后可能有哪些潜在成本和风险？
- 我现在是否应该申请研究生，还是先工作 2 年？

---

## 2. 项目目标

### 2.1 核心目标

构建一个可交互、可追问、可联网、具备上下文治理能力的 AI 决策分析 Agent，帮助用户完成以下两类任务：

1. 单项任务分析
用于分析某个单一决策、计划、项目推进到底可能产生的成本、风险、依赖、收益和失败点。

2. 多选项决策参考
用于对多个候选方案进行相对客观的对比分析，输出适用于不同用户偏好的建议。

### 2.2 非目标

以下内容不作为第一阶段目标：

- 替代专业法律、税务、医疗、签证顾问
- 自动执行外部交易或报名等操作
- 为企业级多人协作流程设计复杂权限系统
- 构建全自动、零确认的“强自治”决策机器人

---

## 3. 用户价值

### 3.1 用户痛点

个人在做重大决策时，常见问题包括：

- 只看到表面成本，忽略长期代价
- 只看短期收益，忽略机会成本
- 缺少结构化思考框架
- 无法判断当前信息是否充分
- 难以客观比较多个方案
- 容易被情绪、从众、单一经验或信息偏差影响

### 3.2 产品价值

本项目为用户提供：

- 结构化拆解：把模糊问题拆成可分析维度
- 交互式补充信息：AI 主动追问关键变量
- 风险前置暴露：提前发现潜在坑点
- 方案对比：明确不同选项的收益、成本和适配条件
- 联网补证：结合公开信息进行外部事实校验
- 图表表达：把趋势、区间、对比关系以可视化方式展示
- 计算辅助：对用户关心的公式、阈值、交点、盈亏平衡点等进行定量求解
- 成本控制：通过上下文治理与缓存优化降低大模型调用费用

## 4. 产品形态

### 4.1 产品类型
一个个人使用的 Agent 型应用。

### 4.2 使用方式
用户先输入一个相对简单的问题，系统先做第一轮轻量理解，再提出问题和预设选项。用户填写后，系统进入“结构化循环分析”流程：持续总结当前已知信息、生成下一轮补充问题、规划联网搜索、沉淀阶段性结论；直到没有新的额外信息需要获取时，再由一个全新上下文的 AI 基于沉淀后的核心结论生成最终 Markdown 报告与图表。

### 4.3 交互模式

- 问答式输入
- 表单式选项补充
- AI 主动追问
- 可拒答 / 跳过
- 可自定义输入
- 循环式补充信息与联网查找
- 最终输出为结构化分析报告 + 图表

### 4.4 核心对话流程原则

- 追问、联网、总结、最终写作分阶段进行，不混在同一个上下文里
- 每一轮都使用“当前已知信息摘要”作为输入，而不是直接拼接全部原始聊天记录
- 搜索使用相对纯净的新上下文执行，避免被冗余对话污染
- 最终报告生成使用全新上下文，只输入沉淀后的主要结论，不输入中间追问草稿

## 5. 核心功能设计

### 5.1 功能总览

MVP 阶段建议包含 7 个核心模块：

1. 分析模式选择器
2. 动态追问引擎
3. 大模型分析引擎
4. Brave 搜索联网模块
5. 图表绘制 MCP 模块
6. 数学计算 MCP 模块
7. 上下文治理与缓存优化模块

### 5.2 分析模式选择器

用户首先选择任务类型：

#### 模式 A：单项任务的成本 / 风险分析
适合分析：

- 是否推进某个计划
- 是否采取某个行动
- 某个计划执行到底的潜在代价

输出重点：

- 成本全景
- 风险全景
- 前提条件
- 风险缓释建议
- 是否值得推进

#### 模式 B：多选项决策参考
适合分析：

- A 和 B 怎么选
- 多个候选路径如何比较
- 不同条件下最优选择是什么

输出重点：

- 选项对比矩阵
- 各选项优劣势
- 对不同偏好下的建议
- 建议结论与适用条件

### 5.3 动态追问引擎

AI 在正式分析前或分析中，如果发现关键信息缺失，需要主动提出追问。

#### 追问目标

- 补全用户约束条件
- 提高分析准确度
- 避免用错误假设得出错误结论

#### 追问示例

- 你目前的收入水平是多少？
- 你家庭是否能承担海外交换的额外支出？
- 你毕业后的目标更偏向升学、就业还是移民？
- 你对风险的接受度更偏保守还是激进？

#### 追问交互要求

每个问题支持：

- 若干推荐选项
- 用户自定义输入
- 用户选择“暂不回答 / 不确定 / 拒绝回答”

#### 设计原则

- 只问高价值问题，不做无意义追问
- 控制追问轮次，避免用户疲劳
- 当用户拒答时，系统应转入“基于不完整信息的保守分析”
- 每个问题都应说明它为什么重要
- 每轮追问必须以结构化结果返回，便于前端直接渲染为表单

#### 追问输出协议

AI 在循环分析阶段返回的“需要用户提供的额外信息”必须是结构化列表，每条至少包含：

- question：问题文本
- purpose：为什么要问
- options：若干预设选项
- allow_custom_input：是否允许自定义输入
- priority：优先级

### 5.4 大模型分析引擎

大模型负责完成推理、拆解、对比和建议输出。

#### 分析能力要求

- 问题结构化拆解
- 成本识别与分类
- 风险识别与分类
- 多维权重分析
- 不确定性标注
- 生成决策建议
- 输出下一步行动建议

#### 成本分析维度

- 金钱成本
- 时间成本
- 机会成本
- 心理成本
- 社交 / 关系成本
- 执行复杂度成本
- 长期维护成本

#### 风险分析维度

- 财务风险
- 学业 / 职业风险
- 时间延误风险
- 信息不对称风险
- 身体 / 心理风险
- 法律 / 政策 / 签证风险
- 外部环境变化风险

#### 决策分析维度

- 与用户目标的匹配度
- 短期收益 vs 长期收益
- 可逆性
- 失败代价
- 可替代性
- 稳定性
- 上升空间

### 5.5 Brave 搜索联网模块

当分析依赖外部事实时，系统需要调用 Brave Search API 进行联网搜索。

#### 联网场景

- 海外交换学校 / 国家生活成本
- 签证政策、奖学金信息
- 某职业薪资区间
- 行业现状与就业趋势
- 某项目所在市场的公开信息

#### 联网原则

- 优先搜索事实，不直接搜索结论
- 将联网结果与模型推理严格分层
- 输出中标注哪些结论来自外部资料，哪些属于模型推断
- 对来源做基础可信度判断
- 搜索任务必须由上游 AI 以结构化协议生成，而不是自由文本描述

#### 搜索任务输出协议

AI 在循环分析阶段返回的“需要联网查找的信息”必须是结构化列表，每条至少包含：

- search_topic：需要查找的主题
- search_goal：查找目的
- search_scope：查找范围，例如地区、时间范围、行业范围
- suggested_queries：建议搜索词
- required_fields：希望提取出的字段
- freshness_requirement：是否要求高时效性

#### 搜索结果摘要协议

搜索模块返回给系统的不是网页原文，而是摘要化结果。每条摘要至少包含：

- title：来源标题
- source_url：出处网址
- source_name：来源名称
- fetched_at：抓取时间
- summary：摘要后的关键信息
- extracted_facts：结构化事实列表
- confidence：可信度评估

### 5.6 图表绘制 MCP 模块

系统需要支持图表绘制类 MCP，用于把分析结果转成用户更容易理解的可视化表达。

#### 目标能力

- 绘制折线图、柱状图、散点图、面积图、雷达图、饼图等基础图表
- 绘制方案对比图、成本分布图、风险热力图、时间线图
- 支持根据结构化分析结果自动生成图表配置
- 支持图表标题、坐标轴、单位、图例、注释等基础元信息
- 支持将图表输出为前端可渲染的数据结构或图片资源

#### 典型场景

- 比较不同方案的总成本与风险评分
- 展示某计划在 6 个月、12 个月、24 个月下的现金流趋势
- 展示“投入增加”与“回报增加”之间的关系曲线
- 展示多个选项在不同维度上的雷达对比

#### 设计要求

- 图表生成必须基于明确数据，不能凭空造值
- 图表中的推断区间、估算区间、真实观测值需要区分展示
- 若数据不足，不强行生成图表，改为提示缺少足够数据

### 5.7 数学计算 MCP 模块

系统需要支持计算类 MCP，用于完成大模型不擅长保证精确性的数值计算、公式求解和几何 / 函数分析。

#### 目标能力

- 四则运算、比例、百分比、复利、贴现、分段求值
- 线性 / 非线性方程求解
- 一元 / 多元函数求值
- 函数交点求解
- 盈亏平衡点计算
- 区间极值、阈值判断、敏感性分析
- 可将计算结果回传给图表模块继续可视化

#### 典型场景

- 计算“买车 vs 公共交通”在第几个月出现成本交点
- 计算海外交换的总预算区间与安全冗余
- 计算副业项目在不同转化率下的盈亏平衡点
- 计算两个收入模型在不同时间下的收益交点

#### 设计要求

- 大模型负责提出计算任务，计算 MCP 负责返回可验证结果
- 对输入参数、单位、范围进行校验
- 对关键计算保留公式、输入参数、结果与误差说明
- 当问题适合公式求解时，优先走计算 MCP 而非纯文本估算

### 5.8 输出报告模块

最终输出建议是结构化报告，便于用户快速阅读和复盘。

#### 单项任务报告建议结构

1. 问题定义
2. 关键假设
3. 成本清单
4. 风险清单
5. 最坏情况 / 最可能情况 / 理想情况
6. 是否值得推进
7. 如果要推进，建议先满足的前置条件
8. 如果暂不推进，替代行动建议
9. 关键计算结果
10. 可视化图表

#### 多选项决策报告建议结构

1. 决策目标
2. 比较维度
3. 选项对比表
4. 各选项优劣势
5. 对不同偏好人群的推荐
6. 当前信息不足点
7. 关键计算结果
8. 图表对比结果
9. 综合建议

### 5.9 阶段性结论沉淀模块

系统在每一轮循环后，都要把阶段性结论沉淀到“主要结论文档”中，供后续轮次和最终报告使用。

#### 沉淀内容

- 用户原始问题的结构化表达
- 用户已回答的问题与答案摘要
- 已完成搜索的摘要结果与出处网址
- 已完成计算的公式、输入与结果
- 当前轮次 AI 得出的新的主要结论

#### 设计原则

- 只保留对最终决策有帮助的高价值结论
- 结论必须区分“事实”“估算”“推断”
- 同类结论需要增量更新，避免重复堆积
- 中间追问文本本身不进入最终报告上下文，只保留其答案与提炼后的结论

## 6. 用户流程设计

### 6.1 主流程

#### 阶段 1：初始问题理解

1. 用户输入一个简单问题
2. AI 基于初始问题做第一轮轻量理解
3. AI 返回第一批问题和预设选项
4. 用户填写这些选项

#### 阶段 2：循环式结构化分析

5. 应用程序总结当前已知信息，形成“当前分析上下文包”
6. 上下文包至少包含：
   - 用户初始问题
   - 用户对所有已提问题的回答
   - 已完成搜索的摘要结果与出处网址
   - 已完成计算的结果
   - 历史轮次沉淀出的主要结论
7. 将该上下文包发送给 AI，要求 AI 返回结构化结果：
   - a. 需要用户提供的额外信息
   - b. 需要联网查找的信息
   - c. 根据现有信息得出的新的总结性主要结论
8. 前端把 a 渲染为用户继续填写的问题表单
9. 搜索系统使用新的纯净上下文执行 b，对每项搜索任务进行联网查找
10. 搜索系统返回摘要化搜索结果，且每条结果必须包含出处网址
11. 系统把 c 补充进“主要结论文档”
12. 系统把新的用户回答、搜索摘要、计算结果一并并入上下文包
13. 如果 a 或 b 仍然非空，则继续下一轮循环
14. 如果 AI 判断没有额外需要获取的信息，则结束循环

#### 阶段 3：最终报告生成

15. 启动一个全新的上下文 AI
16. 只输入之前沉淀出的所有总结性主要结论，不输入中间追问本身
17. AI 输出 Markdown 格式的最终总结文档
18. 如有需要，AI 调用计算 MCP 和图表 MCP 生成折线图、柱状图、雷达图等结果
19. 前端展示最终报告与图表

### 6.2 循环分析阶段的标准输入

每次发给循环分析 AI 的上下文包建议固定为以下结构：

- problem_statement：用户原始问题
- confirmed_user_facts：用户已确认的事实
- answered_questions：用户已回答的问题及答案
- search_summaries：历史搜索结果摘要，且包含出处网址
- calculation_summaries：历史计算结果
- major_conclusions：已沉淀的主要结论
- unresolved_uncertainties：尚未解决的不确定点

### 6.3 循环分析阶段的标准输出

循环分析 AI 必须返回 3 个结构化部分：

#### A. 需要用户提供的额外信息

每项包含：

- question
- purpose
- options
- allow_custom_input
- priority

#### B. 需要联网查找的信息

每项包含：

- search_topic
- search_goal
- search_scope
- suggested_queries
- required_fields
- freshness_requirement

#### C. 新的总结性主要结论

每项包含：

- conclusion
- conclusion_type：事实 / 估算 / 推断
- basis：依据哪些用户信息、搜索结果或计算结果
- confidence：置信度

### 6.4 最终报告阶段的上下文规则

- 使用全新的上下文，不继承循环分析阶段的原始问答记录
- 只输入主要结论文档、必要的计算结果、必要的图表数据
- 不输入“还需要问什么”“还需要搜什么”这类中间控制信息
- 最终报告任务只负责整理、对比、解释与展示，不负责继续发散追问

### 6.5 示例流程：是否参加海外交换

用户输入：
“我是否应该参加大三的海外交换？”

第一轮 AI 可能追问：

- 交换地点或国家是哪里？
- 总预算预计多少？
- 家庭财务支持情况如何？
- 你的长期目标更偏学术、就业还是海外发展？
- 如果不交换，你原本计划做什么？

某一轮 AI 可能要求联网搜索：

- 目标国家生活成本
- 交换学校费用和奖学金
- 签证与停留政策
- 对升学 / 就业可能产生的常见影响

某一轮系统可能执行计算：

- 总预算区间
- 安全冗余金额
- 与“不交换”方案的机会成本差值

最终报告阶段可能绘图：

- 交换 vs 不交换的成本对比柱状图
- 预算区间图
- 多维收益 / 风险雷达图

最后输出：

- 成本分析
- 风险分析
- 机会成本
- 交换 vs 不交换对比
- 不同目标下的建议
- 图表与关键计算结论

## 7. 信息架构与数据结构建议

### 7.1 核心对象

#### AnalysisSession
一次完整分析会话。

建议字段：

- session_id
- user_id
- mode
- problem_statement
- current_status
- created_at
- updated_at

#### ClarificationQuestion
AI 提出的补充问题。

建议字段：

- question_id
- session_id
- question_text
- question_reason
- answer_type
- options
- allow_custom_input
- allow_skip

#### UserAnswer
用户对补充问题的回答。

建议字段：

- answer_id
- question_id
- selected_option
- custom_input
- answer_status

#### EvidenceItem
联网或内部分析中使用的证据块。

建议字段：

- evidence_id
- source_type
- source_url
- title
- snippet
- extracted_fact
- timestamp
- confidence

#### CalculationTask
一次发往计算 MCP 的任务。

建议字段：

- calculation_id
- session_id
- task_type
- formula_expression
- input_params
- units
- result
- error_margin
- created_at

#### ChartArtifact
一次发往图表 MCP 的产物。

建议字段：

- chart_id
- session_id
- chart_type
- data_schema

## 9. 上下文治理设计

本项目的关键工程难点之一，不是“能不能调模型”，而是“如何把正确的信息以最省钱、最稳定的方式送进模型”。

### 9.1 上下文治理目标

- 限制无关信息进入上下文
- 保持 prompt 结构稳定
- 提高缓存命中率
- 降低 token 消耗
- 保证输出可解释

### 9.2 建议的上下文分层

模型调用上下文建议拆成 6 层：

#### 第 1 层：固定系统层

- Agent 身份
- 分析原则
- 输出格式约束
- 风险提示规则

#### 第 2 层：任务模板层

- 单项分析模板
- 多选决策模板

#### 第 3 层：用户已确认事实层

- 收入水平
- 预算范围
- 风险偏好
- 时间约束

#### 第 4 层：外部证据层

- 清洗后的事实块
- 来源
- 时间
- 可信度

#### 第 5 层：计算结果层

- 公式
- 输入参数
- 精确计算结果
- 误差或适用范围说明

#### 第 6 层：当前任务增量层

- 当前要继续追问什么
- 当前要生成什么报告
- 当前要渲染什么图表

### 9.3 主要结论文档原则

主要结论文档是整个系统的核心中间资产。

- 每一轮 AI 只向其中追加“新的总结性主要结论”
- 结论文档要尽量短、稳、可复用
- 后续轮次优先读取结论文档，而不是反复消费完整历史对话
- 最终报告阶段主要依赖结论文档生成

### 9.4 上下文裁剪原则

- 不要把完整聊天记录无脑塞给模型
- 只保留对当前结论有贡献的信息
- 历史轮次优先转成结构化摘要
- 长文本先 extraction，再 injection
- 区分“事实”“假设”“推断”“情绪表达”

### 9.5 纯净上下文策略

- 用户回答补充问题时，前端只提交结构化答案，不提交页面噪音信息
- 联网搜索阶段使用新的纯净上下文，只接收结构化搜索任务
- 最终报告阶段使用新的纯净上下文，只接收主要结论与必要图表数据
- 尽量避免让一个上下文同时承担“提问”“搜索规划”“最终写作”三种角色

## 10. 缓存命中与 API 成本优化

### 10.1 优化目标

- 减少重复 token 输入
- 提升静态 prompt 命中缓存概率
- 控制联网检索、计算结果、图表配置带来的上下文膨胀
- 降低复杂任务多轮调用成本

### 10.2 具体策略

#### 策略 A：稳定系统提示词
系统层 prompt 尽量固定，不频繁改动措辞和顺序。

#### 策略 B：模板化输出要求
分析模式、输出结构、评分维度都使用稳定模板。

#### 策略 C：结构化用户事实
把用户回答转成标准字段，而不是每轮都附带原始对话。

#### 策略 D：证据块标准化
搜索结果统一切分成短小、可复用的 evidence block。

#### 策略 E：计算与图表结果结构化缓存
相同公式、相同参数、相同图表配置优先复用缓存结果。

#### 策略 F：分阶段调用
建议拆成：

1. 问题理解
2. 追问生成
3. 搜索规划
4. 计算执行
5. 图表生成
6. 最终分析

#### 策略 G：主要结论文档优先
优先缓存“主要结论文档”和结构化搜索摘要，而不是缓存大量原始轮次消息。

#### 策略 H：局部重算
当用户只修改一个局部条件时，只重算受影响部分，而不是整份分析全量重跑。

## 11. 联网搜索策略设计

### 11.1 Brave Search API 的角色
Brave Search API 在系统中承担“事实补证”的角色，而不是“替代思考”的角色。

### 11.2 搜索查询生成策略

搜索词应围绕事实变量展开，例如：

- `Japan student exchange living cost 2026`
- `Singapore exchange visa requirements 2026`
- `average internship salary finance undergraduate 2026`

避免直接搜索：

- “我该不该交换”
- “哪个选择更好”

### 11.3 搜索质量控制

- 去重
- 限制结果数量
- 保留来源域名
- 标注抓取时间
- 标注是否属于高时效信息

### 11.4 搜索失败时的降级策略

- 无法联网时，明确提示当前结论缺少外部事实支持
- 输出中标明哪些部分仅基于通用经验推理
- 允许用户补充外部资料后继续分析

## 12. 风险与挑战

### 12.1 产品风险

#### 风险 1：用户把系统当成绝对权威
应对：

- 明确声明是辅助决策工具，不替代专业意见
- 所有结论要带假设前提和不确定性说明

#### 风险 2：用户信息不足导致结论偏差
应对：

- 强化动态追问
- 在报告中标出高影响未知项

#### 风险 3：联网信息质量参差不齐
应对：

- 做来源筛选
- 在输出中标注证据来源和时效

#### 风险 4：图表误导用户
应对：

- 区分真实数据、估算值和推断值
- 图表必须标单位、时间范围、数据来源

#### 风险 5：计算结果被错误理解
应对：

- 关键计算同时展示输入参数、公式和适用条件
- 对超出模型范围的问题返回“不足以精确计算”

### 12.2 技术挑战

- 如何判断“该问什么”而不是“问很多”
- 如何把自然语言压缩成高质量结构化事实
- 如何把搜索结果转成真正有价值的证据块
- 如何稳定 prompt 结构提升缓存命中
- 如何把大模型推理与精确计算可靠协同
- 如何让图表生成既自动化又不失真

## 15. 成功指标

- 搜索结果注入长度受到控制
- 计算与图表调用失败时可平滑降级
- 多轮任务不会因上下文失控而明显劣化
- 主要结论文档长度可控且可复用

## 17. 项目结论

Genius Actuary 的核心不是“更会聊天”，而是“更会做高代价决策分析”。

它应当是一套围绕个人决策问题构建的、具备以下能力的 Agent：

- 会识别缺失信息
- 会主动提问
- 会联网补充事实
- 会做结构化成本与风险分析
- 会做精确计算并给出交点、阈值、盈亏平衡点等结果
- 会把结论转成图表帮助用户理解
- 会在不确定条件下给出有边界的建议
- 会严格控制上下文与调用成本

如果 MVP 做得好，这个项目后续可以自然扩展到：

- 留学 / 职业 / 副业 / 创业等垂直模板
- 用户偏好画像
- 更强的可视化对比能力
- 更完整的历史决策复盘系统
