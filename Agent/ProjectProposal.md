# Genius Actuary 项目策划案

## 1. 项目概述

### 1.1 项目名称
Genius Actuary

### 1.2 项目定位
一个面向个人用户的 AI 精算师 Agent。  
核心目标不是给出“拍脑袋式建议”，而是将一个决策、一项计划、一次行动推进到底后可能产生的：

- 显性成本
- 隐性成本
- 短期风险
- 长期风险
- 机会成本
- 不确定性来源
- 可替代方案
- 决策建议

系统化梳理出来，并通过结构化分析辅助用户做出更理性的判断。

### 1.3 典型场景

- 我是否应该参加大学大三的海外交换？
- 我是否应该辞职创业？
- 我是否应该买车而不是继续打车/坐地铁？
- 这个副业项目推进 6 个月后可能有哪些潜在成本和风险？
- 我现在是否应该申请研究生，还是先工作 2 年？

---

## 2. 项目目标

### 2.1 核心目标

构建一个可交互、可追问、可联网、具备上下文治理能力的 AI 决策分析 Agent，帮助用户完成以下两类任务：

1. 单项任务分析
用于分析某个单一决策、计划、项目推进到底可能产生的成本、风险、依赖、收益和失败点。

2. 多选项决策参考
用于对多个候选方案进行相对客观的对比分析，输出适用于不同用户偏好的建议。

### 2.2 非目标

以下内容不作为第一阶段目标：

- 替代专业法律、税务、医疗、签证顾问
- 自动执行外部交易或报名等操作
- 为企业级多人协作流程设计复杂权限系统
- 构建全自动、零确认的“强自治”决策机器人

---

## 3. 用户价值

### 3.1 用户痛点

个人在做重大决策时，常见问题包括：

- 只看到了表面成本，忽略长期代价
- 只看短期收益，忽略机会成本
- 缺少结构化思考框架
- 无法判断自己当前信息是否充分
- 难以客观比较多个方案
- 容易被情绪、从众、单一经验或信息偏差影响

### 3.2 产品价值

本项目为用户提供：

- 结构化拆解：把模糊问题拆成可分析维度
- 交互式补充信息：AI 主动追问关键变量
- 风险前置暴露：提前发现潜在坑点
- 方案对比：明确不同选项的收益、成本和适配条件
- 联网补证：结合公开信息进行外部事实校验
- 成本控制：通过上下文治理与缓存优化降低大模型调用费用

## 4. 产品形态

### 4.1 产品类型
一个个人使用的 Agent 型应用。

### 4.2 使用方式
用户进入系统后，先选择分析模式，再输入问题，系统通过多轮交互收集必要信息，随后调用大模型和联网搜索能力完成分析，并以结构化报告输出结果。

### 4.3 交互模式

- 问答式输入
- 表单式选项补充
- AI 主动追问
- 可拒答/跳过
- 可自定义输入
- 最终输出为结构化分析报告

## 5. 核心功能设计

## 5.1 功能总览

MVP 阶段建议包含 5 个核心模块：

1. 分析模式选择器
2. 动态追问引擎
3. 大模型分析引擎
4. Brave 搜索联网模块
5. 上下文治理与缓存优化模块

### 5.2 分析模式选择器

用户首先选择任务类型：

#### 模式 A：单项任务的成本/风险分析
适合分析：

- 是否推进某个企划
- 是否采取某个行动
- 某个计划执行到底的潜在代价

输出重点：

- 成本全景
- 风险全景
- 前提条件
- 风险缓释建议
- 是否值得推进

#### 模式 B：多选项决策参考
适合分析：

- A 或 B 怎么选
- 多个候选路径如何比较
- 不同条件下最佳选择是什么

输出重点：

- 选项对比矩阵
- 各选项优劣势
- 对不同偏好下的建议
- 建议结论与适用条件

### 5.3 动态追问引擎

AI 在正式分析前或分析中，如果发现关键信息缺失，需要主动提出追问。

#### 追问目标

- 补全用户约束条件
- 提高分析准确度
- 避免用错误假设得出错误结论

#### 追问示例

- 你目前的收入水平是多少？
- 你家庭是否能承担海外交换的额外支出？
- 你毕业后的目标更偏向升学、就业还是移民？
- 你对风险的接受度更偏保守还是激进？

#### 追问交互要求

每个问题支持：

- 若干推荐选项
- 用户自定义输入
- 用户选择“暂不回答 / 不确定 / 拒绝回答”

#### 设计原则

- 只问高价值问题，不做无意义追问
- 控制追问轮次，避免用户疲劳
- 当用户拒答时，系统应转入“基于不完整信息的保守分析”
- 每个问题都应说明它为什么重要

### 5.4 大模型分析引擎

大模型负责完成推理、拆解、对比和建议输出。

#### 分析能力要求

- 问题结构化拆解
- 成本识别与分类
- 风险识别与分类
- 多维权衡分析
- 不确定性标注
- 生成决策建议
- 输出下一步行动建议

#### 成本分析维度

- 金钱成本
- 时间成本
- 机会成本
- 心理成本
- 社交/关系成本
- 执行复杂度成本
- 长期维护成本

#### 风险分析维度

- 财务风险
- 学业/职业风险
- 时间延误风险
- 信息不对称风险
- 身体/心理风险
- 法律/政策/签证风险
- 外部环境变化风险

#### 决策分析维度

- 与用户目标的匹配度
- 短期收益 vs 长期收益
- 可逆性
- 失败代价
- 可替代性
- 稳定性
- 上升空间

### 5.5 Brave 搜索联网模块

当分析依赖外部事实时，系统需要调用 Brave Search API 进行联网搜索。

#### 联网场景

- 海外交换学校/国家生活成本
- 签证政策、奖学金信息
- 某职业薪资区间
- 行业现状与就业趋势
- 某项目所在市场的公开信息

#### 联网原则

- 优先搜索事实，不直接搜索结论
- 将联网结果与模型推理严格分层
- 输出中标注哪些结论来自外部资料，哪些属于模型推断
- 对来源做基础可信度判断

#### 搜索结果处理流程

1. 根据分析任务生成搜索子问题
2. 调用 Brave Search API 检索
3. 过滤无关、低质量、重复信息
4. 提取可复用事实块
5. 将事实块注入分析上下文
6. 在最终报告中标注引用来源与适用时间

### 5.6 输出报告模块

最终输出建议是结构化报告，便于用户快速阅读和复盘。

#### 单项任务报告建议结构

1. 问题定义
2. 关键假设
3. 成本清单
4. 风险清单
5. 最坏情况 / 最可能情况 / 理想情况
6. 是否值得推进
7. 如果要推进，建议先满足的前置条件
8. 如果暂不推进，替代行动建议

#### 多选项决策报告建议结构

1. 决策目标
2. 比较维度
3. 选项对比表
4. 各选项优劣势
5. 对不同偏好人群的推荐
6. 当前信息不足点
7. 综合建议

## 6. 用户流程设计

### 6.1 主流程

1. 用户进入应用
2. 选择分析模式
3. 输入问题描述
4. 系统抽取任务类型、目标、约束、未知信息
5. AI 发起动态追问
6. 用户回答、跳过或自定义补充
7. 系统判断是否需要联网搜索
8. 执行 Brave 搜索并整理事实块
9. 调用大模型生成分析结果
10. 输出结构化报告
11. 用户可继续追问、调整假设或重跑分析

### 6.2 示例流程：是否参加海外交换

用户输入：
“我是否应该参加大三的海外交换？”

系统可能追问：

- 交换地点或国家是哪里？
- 总预算预计多少？
- 家庭财务支持情况如何？
- 你的长期目标更偏学术、就业还是海外发展？
- 如果不交换，你原本计划做什么？

系统联网搜索：

- 目标国家生活成本
- 交换学校费用和奖学金
- 签证与停留政策
- 对升学/就业可能产生的常见影响

最后输出：

- 成本分析
- 风险分析
- 机会成本
- 交换 vs 不交换 对比
- 在不同目标下的建议

## 8. 系统架构建议

### 8.1 架构原则

- 模块化
- 可替换模型提供方
- 上下文可控
- 联网模块与推理模块解耦
- 对缓存友好
- 对审计和复盘友好

### 8.2 推荐模块划分

#### 1. 前端交互层
负责：

- 模式选择
- 问题输入
- 追问表单展示
- 分析报告展示

#### 2. Orchestrator 编排层
负责：

- 判断当前阶段
- 决定是否继续追问
- 决定是否联网
- 构建最终模型请求
- 控制成本与缓存

#### 3. Clarification Engine
负责：

- 识别缺失变量
- 生成追问问题
- 控制追问优先级与轮次

#### 4. Search Adapter
负责：

- 对接 Brave Search API
- 查询重写
- 结果去重与清洗
- 输出标准化 evidence

#### 5. Prompt & Context Manager
负责：

- 固定系统提示词
- 上下文分层
- 截断策略
- 缓存友好的 prompt 拼装

#### 6. Analysis Engine
负责：

- 调用大模型
- 执行结构化分析
- 生成最终报告

#### 7. Persistence Layer
负责：

- 保存会话
- 保存追问与回答
- 保存检索证据
- 保存报告版本

## 9. 上下文治理设计

本项目的关键工程难点之一，不是“能不能调模型”，而是“如何把正确的信息以最省钱、最稳定的方式送进模型”。

### 9.1 上下文治理目标

- 限制无关信息进入上下文
- 保持 prompt 结构稳定
- 提高缓存命中率
- 降低 token 消耗
- 保证输出可解释

### 9.2 建议的上下文分层

模型调用上下文建议拆成 5 层：

#### 第 1 层：固定系统层
内容相对稳定，例如：

- Agent 身份
- 分析原则
- 输出格式约束
- 风险提示规则

该层应尽量长期稳定，以提高 prompt caching 命中率。

#### 第 2 层：任务模板层
根据模式不同加载不同模板，例如：

- 单项分析模板
- 多选决策模板

这一层也尽量保持稳定、标准化。

#### 第 3 层：用户已确认事实层
只放用户明确输入或明确选择过的信息，例如：

- 收入水平
- 预算范围
- 风险偏好
- 时间约束

避免混入模糊自然语言原文，优先用结构化摘要。

#### 第 4 层：外部证据层
只放经过清洗后的事实块，不直接塞入大段网页原文。

每条 evidence 最好包含：

- 事实内容
- 来源
- 时间
- 可信度

#### 第 5 层：当前任务增量层
只放这一次调用真正需要的增量指令，例如：

- 当前要继续追问
- 当前要生成最终报告
- 当前要比较两个选项

### 9.3 上下文裁剪原则

- 永远不要把完整聊天记录无脑塞给模型
- 只保留对当前结论有贡献的信息
- 历史轮次优先转成结构化摘要
- 长文本先做 extraction，再做 injection
- 区分“事实”、“假设”、“推断”、“情绪表达”

### 9.4 建议的会话记忆策略

短期记忆：
仅服务当前分析任务。

长期记忆：
仅在用户授权下保留偏好，例如：

- 风险偏好
- 财务保守程度
- 常用决策标准

敏感信息默认不做长期保存。

## 10. 缓存命中与 API 成本优化

这是本项目非常重要的工程目标。

### 10.1 优化目标

- 减少重复 token 输入
- 提升静态 prompt 命中缓存概率
- 控制联网检索带来的上下文膨胀
- 降低复杂任务多轮调用成本

### 10.2 具体策略

#### 策略 A：稳定系统提示词
系统层 prompt 尽量固定，不频繁变动措辞和顺序。

#### 策略 B：模板化输出要求
分析模式、输出结构、评分维度都使用稳定模板。

#### 策略 C：结构化用户事实
把用户回答转成标准字段，而不是每轮都附带原始对话。

示例：

不推荐：
“用户之前说自己家里大概能支持一些，但不想花太多。”

推荐：

- family_support_level: moderate
- spending_sensitivity: high

#### 策略 D：证据块标准化
搜索结果统一切分成短小、可复用的 evidence block。

#### 策略 E：分阶段调用
不要一次调用承担所有事情。

建议拆成：

1. 问题理解
2. 追问生成
3. 搜索规划
4. 最终分析

这样既能控成本，也更利于缓存复用。

#### 策略 F：结果复用
当用户只是修改一个局部条件时，只重新计算受影响部分。

例如：

- 预算变化
- 国家变化
- 时间约束变化

而不是整个分析全量重跑。

### 10.3 需要特别关注的成本来源

- 长系统提示词
- 冗余历史消息
- 原始网页大段注入
- 一次性要求模型完成过多任务
- 频繁使用不同措辞导致缓存失效

## 11. 联网搜索策略设计

### 11.1 Brave Search API 的角色

Brave Search API 在系统中承担“事实补证”角色，而不是“替代思考”的角色。

### 11.2 搜索查询生成策略

搜索词应围绕事实变量展开，例如：

- `Japan student exchange living cost 2026`
- `Singapore exchange visa requirements 2026`
- `average internship salary finance undergraduate 2026`

避免直接搜索：

- “我该不该交换”
- “哪个选择更好”

### 11.3 搜索质量控制

- 去重
- 限制结果数量
- 保留来源域名
- 标注抓取时间
- 标注是否为高时效信息

### 11.4 搜索失败时的降级策略

- 无法联网时，明确提示当前结论缺少外部事实支持
- 输出中标明哪些部分仅基于通用经验推理
- 允许用户补充外部资料后继续分析

## 12. 风险与挑战

### 12.1 产品风险

#### 风险 1：用户把系统当成绝对权威
应对：

- 明确声明是辅助决策工具，不替代专业意见
- 所有结论要带假设前提和不确定性说明

#### 风险 2：用户信息不足导致结论偏差
应对：

- 强化动态追问
- 在报告中标出高影响未知项

#### 风险 3：联网信息质量参差不齐
应对：

- 做来源筛选
- 在输出中标注证据来源和时效

#### 风险 4：API 成本快速上升
应对：

- 严格上下文治理
- 分阶段调用
- 结构化缓存策略

### 12.2 技术挑战

- 如何判断“该问什么”而不是“问很多”
- 如何把用户自然语言压缩成高质量结构化事实
- 如何将搜索结果转成真正有价值的证据块
- 如何稳定 prompt 结构提升缓存命中
- 如何平衡分析深度与响应成本

## 13. MVP 范围建议

### 13.1 MVP 目标

先验证这件事是否真的能帮助用户提升决策质量，并验证“动态追问 + 联网补证 + 结构化输出”的核心闭环。

### 13.2 MVP 必须具备的能力

- 支持两种模式：
  - 单项成本/风险分析
  - 多选项决策参考
- 支持用户输入问题
- 支持 AI 动态追问
- 支持选项式回答 + 自定义输入 + 跳过
- 支持 Brave Search API 基础联网
- 支持结构化分析报告输出
- 支持基础上下文治理
- 支持基础日志与调用追踪

### 13.3 MVP 暂不包含

- 用户账户体系复杂权限
- 长期记忆人格化
- 多模型路由优化
- 复杂可视化图表系统
- 移动端原生应用
- 自动化行动执行

## 17. 项目结论

Genius Actuary 的核心不是“更会聊天”，而是“更会做高代价决策分析”。  
它应当是一套围绕个人决策问题构建的、具备以下能力的 Agent：

- 会识别缺失信息
- 会主动提问
- 会联网补充事实
- 会做结构化成本与风险分析
- 会在不确定条件下给出有边界的建议
- 会严格控制上下文与调用成本

如果 MVP 做得好，这个项目后续可以自然扩展到：

- 留学/职业/副业/创业等垂直模板
- 用户偏好画像
- 更强的可视化对比能力
- 更完整的历史决策复盘系统
